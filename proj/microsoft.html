Challenge

Creating a more intuitive chatbot builder for business administrators to build simple flows.

Outcome

A redesigned, drag-and-drop builder with a new visual identity and user experience. Jump to the final designs here!

Impact

Overview

I have permission from Microsoft to showcase this project. A free demo can be requested when you contact our customer engineers at cloudcrew@microsoft.com.

Internal operational communication is time-consuming. Therefore, we designed Bot Portal, a streamlined chatbot builder that is more compact than Power Virtual Agents, a web app that builds complex chatbot conversation flows.

As the only designer on the team, I led the web app's end-to-end UI/UX redesign. I was partnered with another senior engineer who acted as the PM and provided me with requirements to validate. Ultimately, some of my designs may only serve as a visionary demonstration, but the essential features are set to be launched in Q4 2021.

Problem

How do we make the chatbot builder accessible for users to create simple flows?

Before diving into the process, let me clarify some keywords that will explain design choices going forward:

Pain Points

Through user interviews, the senior engineer and I discovered an operations manager's top difficulties when building a conversation flow.

  1. Unintuitive interaction: The user must build flows by creating new if/or/and statements and use indices to represent multiple choices. Tldr. It's too hard to use.
  2. Interrupted flows: Text and file editors all exist as modals currently, which often interrupts the user's actions and disables the main content.
  3. Choice overload: There are 14 response nodes for users to choose from, easily overwhelming users building simple flows with less than five responses.
  4. Redundant workflows: It is time-consuming to recreate the same nodes.

Project Goals

So, how do we streamline the chatbot-building experience?

  1. Create intuitive flow-building interactions
  2. Prioritize frequent actions and restructure information architecture
  3. Enable users to save common nodes

Solution

The key features are summarized in the Final Designs section. Jump to the final designs here!

Goal 1: Create intuitive interactions

Tree Structure

The current structure imitates code, requiring the user to initiate loops and add if statements. Referring to user feedback that wants a clearer display, I changed it to a vertical tree diagram to better show the sequence of events.

Before: Code structure

After: Tree structure - clearer display of the same content on the left

Drag and Drop Interaction

The current implementation requires users to use small add buttons to connect a response node. However, according to Fitt's law, the small targets are difficult for fast, frequent actions.

Therefore, by performing competitive analysis on similar products, the senior engineer and I decided on a drag-and-drop interaction. Users can build a conversation flow with little to no training faster.

Before: Small add buttons

After: Drag and drop

Sidebar

To reduce the use of modals that blocks information on the screen, I introduced the sidebar. It effectively hides complicated actions and displays more options on one screen.

Before: Modal

After: Sidebar - condenses the same content on the left

Goal 2: Prioritize frequent actions

Reducing Response Nodes

Through card sorting, I categorized actions under shared user motivations, which helped me combine response nodes. I reduced the number of choices by 50%, from 14 to 7, with the most frequently used "Show a message" node at the top.

Before: 14 choices - too many!

After: 7 choices

Goal 3: Save common nodes

Copy and Paste vs Global nodes

The current implementation copies and pastes a response node directly into the same intent, but user feedback states that they also want to paste into other intents and edit all copies of the same node at once.

Thus, I came up with the "Copy to clipboard" idea, allowing users to save common nodes to a clipboard that is accessible across all intents. However, user feedback shows that there are details in common nodes that are often updated, such as system version ids when the user asks the chatbot about IT-related topics.

Consequently, I prototyped a "Global nodes" feature, where users can save nodes as global nodes. When they edit the master instance, all instances of this node in different intents are also updated. This is the iteration approved by the senior engineer!

Before: Copy and paste into the same intent

1st Iteration: Copy to clipboard, accessible to all intents

2nd Iteration: Global Nodes, editing master component can change instances

Exploring and Testing

I created multiple interactive prototypes to test edge cases and compound use cases. This guided our critiques and evaluation process, which also prompted new features that the senior engineer plan to include in future updates.

New Feature 1: Adding customized intents, essentially creating new words for the bot to recognize through the front-end

New Feature 2: Storing multiple languages in each node, so that the Chatbot responds in the user question's language

Visual Design

To underscore Microsoft's brand identity, my design system adheres to Microsoft's Fluent Design core principles. I also provided a UI Style Guide that lays out all the components and their states to guide the development process in Angular.

Final Designs

Redesign 1

Build flows quickly

Users can simply drag and drop nodes from the toolbox to build the conversation flow.

Redesign 2

Redesign 3

Impact

Despite facing many technical and communication challenges, I was super proud of leading the design of this new iteration, prototyping almost 200 screens!

Takeaways

I grew tremendously as a designer, communicator, and team player during this internship. Some key lessons include:

1. Accessibility

Drag and drop suggests that the user must be able to use a mouse and visually perceive draggable elements and drop zones. To make it accessible, the senior engineer and I discussed a list of requirements- namely, we must make 1) all draggable elements keyboard focusable 2) add event handlers for grabbing, moving, and dropping elements 3) add text that is only visible to the screen reader to guide the grab/move/drop process.

2. Presentation and documentation

While I was only collaborating with one other engineer, I needed to ask questions concisely to make sure my design choices were clear because we mostly worked remotely. Since this was a major redesign, I saw the necessity to write a PRD so a new designer can easily pick up where I finished.

3. Getting feedback early

Demonstrating the functionality through prototypes better represented the user flows I was envisioning, which made the critique process easier. I also sought out designers on other teams to analyze the new user's experience to create informed and guided interactions.
